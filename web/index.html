Faultline

Production breaks. On-call wastes 10 minutes finding which service is at fault and which commit caused it. Faultline does it in 60 seconds.

step tool call result rollback staged final

The idea

When a service breaks, the on-call's first ten minutes are mechanical: which service is the real root, which commit caused it, where is the rollback. Faultline automates that triage. It runs a fixed 8-step investigation policy, reads Google Cloud telemetry + GitLab commits, names the offending commit, drafts the postmortem, and stages a DRAFT rollback merge request. Then it stops. You review and click Approve. The merge fires GitLab CI and the service recovers.

Reasoning: Gemini 2.5 Flash on Vertex AI. Orchestration: Google ADK. GitLab integration: load-bearing MCP server (`@zereight/mcp-gitlab` via stdio). Observability: Cloud Logging / Trace / Monitoring. See the policy →

The flow

  1. Pick alerting service + scenario hint on the left.
  2. Start investigation. Agent runs 8-step policy live. Cards stream in. ~30-90s.
  3. Agent finds the suspect commit, opens GitLab issue, stages DRAFT rollback MR.
  4. You review the issue + MR (links appear in the rollback card).
  5. Approve. Faultline merges the MR via GitLab. Service recovers.

The agent never merges on its own. A human always clicks Approve.
